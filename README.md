# WARNY-BI

WARNY-BI is a DS545 term project that uses retrieval-augmented generation
(RAG) and Power BI to answer vehicle dashboard warning-light questions.

The dashboard is meant to help a driver or service-oriented user understand:

- what the warning light may indicate
- how urgent the case is
- whether recall evidence appears relevant
- what service path is recommended
- which retrieved evidence supports the answer
- what users are asking most often

WARNY-BI is not a certified diagnostic tool. It provides evidence-grounded
triage guidance and should still point users to VIN lookup, owner manuals, OEM
guidance, and professional inspection when needed.

## Quick Start

Most Power BI Desktop users should start with
[`Azure + Power BI Desktop Setup`](#azure--power-bi-desktop-setup). That section
assumes the Azure OpenAI + Azure AI Search RAG path already works in Foundry Chat
Playground and explains how to connect it to Power BI.

For free prompt testing, revise the `.pbix` dashboard file, sync it through
GitHub, then send the prompt to the project maintainer. The maintainer will run

## Pipeline Overview

The Azure submission pipeline is:

```text
processed CSV files
-> Azure Blob Storage
-> Azure SQL Database
-> dbo.vw_rag_documents
-> Azure AI Search
-> Azure OpenAI GPT-4o with Azure AI Search data source
-> Azure Logic App
-> Power Query
-> Power BI dashboard
```

The Logic App is the orchestration layer. Power BI sends one natural-language
prompt to the Logic App. The Logic App calls Azure OpenAI, lets Azure OpenAI use
Azure AI Search as the retrieval source, writes one interaction-log row to Azure
SQL, and returns the Azure OpenAI response to Power BI.

Page 3 reads the SQL log views:

```text
dbo.vw_query_log_dashboard
dbo.vw_query_log_evidence
```

## Azure + Power BI Desktop Setup

Use this section when these Azure pieces already exist:

- Azure SQL Server and Azure SQL Database
- `dbo.vw_rag_documents` loaded with WARNY-BI RAG records
- Azure AI Search service and index
- Azure OpenAI GPT-4o deployment
- Foundry Chat Playground test that returns a structured JSON answer
- Azure Logic App HTTP trigger that calls Azure OpenAI

### Run The SQL Log Script

In DBeaver, connect to the Azure SQL database and run:

```text
scripts/sql/05_create_query_log.sql
```

This creates:

- `dbo.query_log`
- `dbo.vw_query_log_dashboard`
- `dbo.vw_query_log_evidence`

There is no `06` logger script. Logging is done by the Logic App inserting one
row directly into `dbo.query_log`.

### Configure Logic App Logging

In the Logic App, after the Azure OpenAI HTTP action succeeds, add an Azure SQL
**Insert row** action for table `dbo.query_log`.

Set these fields:

| SQL column | Logic App value |
| --- | --- |
| `pipeline` | `azure_logic_app` |
| `user_prompt` | `triggerBody()?['query']` |
| `answer_json` | `body('<Azure OpenAI action>')?['choices'][0]['message']['content']` |
| `citations_json` | `string(body('<Azure OpenAI action>')?['choices'][0]['message']['context']['citations'])` |
| `azure_response_json` | `string(body('<Azure OpenAI action>'))` |

Replace `<Azure OpenAI action>` with the actual action name in your Logic App.
Leave `query_id` and `created_at_utc` blank so Azure SQL uses the table
defaults.

The Logic App's final HTTP response should return the Azure OpenAI HTTP action
body. Power Query expects that raw Azure OpenAI response shape.

### Local Power BI Secrets

Do not store the Logic App trigger URL or Azure SQL connection values directly
in a committed `.pbix`. The Logic App URL contains a `sig=` token and should be
treated as a secret.

Copy the example file:

```text
config/powerbi.secrets.example.json
```

to a local untracked file:

```text
config/powerbi.secrets.json
```

Fill in the real values locally:

```json
{
  "AzureLogicAppUrl": "https://...logic.azure.com/...&sig=...",
  "AzureSqlServer": "freewarnybisqlserver.database.windows.net",
  "AzureSqlDatabase": "free-warny-bi-sql-database"
}
```

The real `config/powerbi.secrets.json` file is ignored by Git.

### Power BI Parameters

Create these parameters in **Transform data** -> **Manage Parameters**:

| Parameter | Type | Example |
| --- | --- | --- |
| `BasePrompt` | Text | `2020 Hyundai Elantra yellow engine light recall` |
| `PowerBIProjectRoot` | Text | Local repository path, for example `C:\Users\you\vscode\warny-bi`. |
| `AzureTopK` | Whole Number | `5` |
| `AzureIncludeImages` | True/False | `false` |

When Power BI asks for credentials for the Logic App URL, use **Anonymous**.
When Power BI asks for Azure SQL credentials, use the same authentication method
that works in DBeaver.

### Power Query Files

Create these Power BI queries:

| Query name | Source file | Enable load? | Purpose |
| --- | --- | --- | --- |
| `LocalConfig` | `src/powerbi/local_config.m` | No | Reads local untracked `config/powerbi.secrets.json` under the repository root. |
| `Query` | `src/powerbi/azure_query.m` | No | Calls the Logic App and normalizes the Azure OpenAI response. |
| `LogQuery` | `src/powerbi/log_query.m` | No | Reads Azure SQL log views for Page 3. |
| `Response` | create manually | No | Calls `Query(...)` using parameters. |
| `Answer` | `src/powerbi/answer_query.m` | Yes | Page 1/2 answer fields. |
| `Evidence` | `src/powerbi/evidence_query.m` | Yes | Page 2 current-response evidence fields. |
| `QueryLog` | create manually | Yes | Page 3 interaction-log table. |
| `QueryLogEvidence` | create manually | Yes | Page 3 logged citation/evidence table. |

For each source file, create a blank query, open **Advanced Editor**, paste the
file contents, and rename the query to the name shown above.

### Response Query

Create a blank query named `Response`:

```powerquery
let
    Config = LocalConfig(PowerBIProjectRoot),
    Source = Query(
        BasePrompt,
        Config[AzureLogicAppUrl],
        AzureTopK,
        AzureIncludeImages
    )
in
    Source
```

Disable **Enable load** for `Response`.

### Dashboard Tables

Create `Answer` from `src/powerbi/answer_query.m` and `Evidence` from
`src/powerbi/evidence_query.m`. Enable load for both.

### Page 3 Log Tables

Create `QueryLog`:

```powerquery
let
    Config = LocalConfig(PowerBIProjectRoot),
    Source = LogQuery(Config[AzureSqlServer], Config[AzureSqlDatabase], "vw_query_log_dashboard", "dbo")
in
    Source
```

Create `QueryLogEvidence`:

```powerquery
let
    Config = LocalConfig(PowerBIProjectRoot),
    Source = LogQuery(Config[AzureSqlServer], Config[AzureSqlDatabase], "vw_query_log_evidence", "dbo")
in
    Source
```

Enable load for both.

Use `QueryLog` for Page 3 visuals such as query volume, frequent makes/models,
frequent warning lights, severity distribution, recall status distribution, and
high-impact query counts.

Use `QueryLogEvidence` for citation/evidence detail visuals such as evidence
count, citation preview, source URL, campaign ID, and warning-light ID.

## First Azure Test

Set:

```text
BasePrompt = 2020 Hyundai Elantra yellow engine light recall
```

Then refresh previews in this order:

1. `Response`: should show a record with `query`, `answer`, `evidence`, and `raw`.
2. `Answer`: should show one row with summary, severity, recall, and parsed vehicle fields.
3. `Evidence`: should show retrieved evidence/citation rows.
4. `QueryLog`: should show a new logged interaction after the Logic App insert succeeds.
5. `QueryLogEvidence`: should show citation rows expanded from `citations_json`.

If `Response` fails, check the Logic App trigger URL and the Azure OpenAI HTTP
action. If `Response` works but Page 3 is empty, check the Logic App SQL
**Insert row** action and confirm that `dbo.query_log` has new rows in DBeaver.

## Free Prompt Testing

The no-cost test backend is maintained by the project maintainer only. Other
users do not need to install or run it.

To test a prompt without spending Azure credits:

1. Edit the Power BI Desktop dashboard file.
2. Sync the revised `.pbix` through GitHub.
3. Send the exact prompt to the maintainer.
4. The maintainer will run the prompt for free and share the returned
   answer/evidence result.

## Preprocessing

Raw CSV files are the local source of truth. The preprocessing script validates
the raw files, creates EDA reports, generates `data/processed/schema.json`, and
writes SQL-ready CSV files.

```bash
python3 scripts/python/preprocess_dataset.py validate
python3 scripts/python/preprocess_dataset.py eda
python3 scripts/python/preprocess_dataset.py schema --csv-dir data/raw --output-file data/processed/schema.json
python3 scripts/python/preprocess_dataset.py clean --schema-file data/processed/schema.json
```
